# Architecture

LOKA runs AI agent workloads and deploys applications in microVMs. On macOS, VMs use the Apple Virtualization Framework. On Linux, they use Cloud Hypervisor with KVM. This page describes the system components and how they interact.

## Components

### Control Plane (CP)

The central API server. Handles session lifecycle, service deployment, scheduling, policy evaluation, checkpoint management, domain routing, and volume management. Stateless -- all persistent state lives in the database and object store.

### Service Manager

Manages deployed applications. Handles multi-component services defined in `loka.yaml` or Docker Compose files, tracks service health, and manages redeployments.

### Domain Proxy

Routes incoming HTTP/HTTPS traffic to the correct service or session based on domain name. Provides automatic `.loka` domains for all services and supports custom domain mappings.

### DNS Server

Built-in DNS server that resolves `.loka` domains to the control plane. Enabled via `loka dns enable`, which configures the system resolver to forward `.loka` queries to LOKA.

### Volume Manager

Manages persistent volume metadata in the control plane. The control plane stores no volume data -- actual data lives on workers (block volumes) or object storage (object volumes). The volume manager handles worker placement, replication assignment, failover on worker death, and degraded volume healing.

Block volumes are replicated across 2 workers. The scheduler prefers workers already holding a volume's primary copy (volume affinity).

### Lock Manager

Coordinates distributed file locking for shared block volumes. Supports both exclusive and shared (multi-holder) locks with per-holder TTLs. Workers acquire locks via the control plane API before writing to shared volumes.

### Scheduler

Assigns sessions and services to worker nodes based on available capacity. Supports affinity hints including volume affinity (prefer workers holding volume primaries) and region/provider preferences.

### Worker

A host machine that runs microVMs. On macOS, the worker uses the Apple Virtualization Framework. On Linux, it uses Cloud Hypervisor with KVM. Each worker runs a VM manager that handles the lifecycle of VMs on that host.

### Supervisor

Runs inside each microVM as PID 1. It starts and stops processes, enforces command policies, manages the approval queue, handles checkpoint I/O, and provides the PTY for shell access.

### MicroVM

Each session or service instance runs in a dedicated microVM with its own kernel, filesystem, and network namespace. On macOS, VMs are managed by the Apple Virtualization Framework. On Linux, VMs are managed by Cloud Hypervisor.

### Snapshot Store

Stores base images, warm snapshots (memory + disk), and checkpoint overlays. Can be backed by local disk or shared storage (NFS, S3).

### Volume Replication

Handles worker-to-worker data replication for block volumes. The primary worker pushes changes to replica workers via HTTP using manifest-based diffing (SHA256 per file). Replication is fully async -- writes are never blocked. Each worker also serves volume data to peers via an HTTP API for initial sync and periodic reconciliation.

## Data flow

1. **SDK/CLI** calls `create_session` or `deploy` on the **CP**.
2. **CP** asks the **Scheduler** for a worker with capacity.
3. **Scheduler** returns a worker. **CP** sends a `StartVM` request to that worker.
4. Worker starts a microVM (VZ on macOS, Cloud Hypervisor on Linux). Session is `running`.
5. **SDK/CLI** calls `run` on the **CP**. **CP** forwards to the **Supervisor** inside the VM.
6. **Supervisor** checks the command against the **policy engine**. If approved, it executes inside the VM.
7. Result flows back: **VM** --> **Supervisor** --> **CP** --> **SDK/CLI**.
8. For services, the **Domain Proxy** routes incoming HTTP traffic to the correct VM based on the `.loka` domain.

## Key files

| Path | Description |
|---|---|
| `cmd/lokad/` | Control plane entry point |
| `cmd/loka/` | CLI client |
| `internal/controlplane/` | CP business logic (sessions, services, scheduling) |
| `internal/controlplane/api/` | HTTP/gRPC API server, domain proxy, volume endpoints |
| `internal/controlplane/service/` | Service manager |
| `internal/controlplane/volume/` | Volume manager (metadata-only, no data on CP) |
| `internal/controlplane/lock/` | Distributed file lock manager (shared + exclusive) |
| `internal/supervisor/` | VM-side command execution, policy enforcement |
| `internal/worker/` | Worker agent, VM lifecycle |
| `internal/worker/volsync/` | Volume sync agent (fsnotify, manifest-based diffing) |
| `internal/worker/volrepl/` | Worker-to-worker volume replication (HTTP) |
| `proto/` | gRPC service definitions |
| `sdk/python/` | Python SDK source |
| `sdk/typescript/` | TypeScript SDK source |
