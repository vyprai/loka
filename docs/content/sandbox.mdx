# Sandbox Architecture

The sandbox is a defense-in-depth system with six enforcement layers.

## Enforcement Layers

| Layer | Component | Purpose |
|---|---|---|
| 1. Command Proxy | Supervisor | Binary gate -- only approved binaries can run |
| 2. Binary Removal | Image build | Dangerous binaries (`mount`, `nsenter`, `chroot`) removed from rootfs |
| 3. Read-Only Filesystem | Overlay mount | Base rootfs mounted RO; writes go to overlay |
| 4. Network Filter | iptables + DNS proxy | Outbound/inbound traffic restricted per policy |
| 5. seccomp | Kernel filter | Syscall filtering on the spawned process |
| 6. cgroups v2 | Kernel | Resource limits: memory, CPU, PIDs, I/O bandwidth |

## seccomp Profiles

### `readonly`

Blocks syscalls that modify system state:

| Blocked Syscalls | Reason |
|---|---|
| `fork`, `clone`, `vfork` | No subprocess creation |
| `unlink`, `unlinkat`, `rmdir` | No file deletion |
| `connect`, `sendto`, `sendmsg` | No outbound networking |
| `mount`, `umount2` | No filesystem manipulation |
| `kill`, `tkill` | No signal sending |

### `standard` (default)

Allows normal process behavior, blocks privilege escalation:

| Blocked Syscalls | Reason |
|---|---|
| `mount`, `umount2` | No filesystem remount |
| `reboot` | Prevent VM reboot |
| `kexec_load` | No kernel replacement |
| `pivot_root`, `chroot` | No root escape |
| `ptrace` | No process tracing |
| `personality` | No ABI change |

### `full`

Minimal block list -- only `reboot`, `kexec_load`, `mount`. Used for trusted workloads.

## What CAN Be Bypassed

These layers can be circumvented by a sufficiently capable process **within the VM**:

- iptables rules (if the process has `CAP_NET_ADMIN` -- it should not).
- Binary gate (if a permitted interpreter like `python3` downloads and runs arbitrary code).
- Filesystem rules (if seccomp-notify has a TOCTOU race -- mitigated by Landlock).

## What CANNOT Be Bypassed

These are enforced by the Firecracker VMM or the host kernel, outside the guest:

- **VM boundary** -- Firecracker's memory isolation. The guest cannot access host memory.
- **cgroups limits** -- enforced by the host kernel on the Firecracker process.
- **Overlay read-only base** -- the base rootfs block device is presented as read-only by Firecracker.
- **vsock-only communication** -- no shared filesystem or network namespace with the host.
- **Jailer** -- Firecracker jailer applies host-side seccomp, chroot, and UID isolation.
