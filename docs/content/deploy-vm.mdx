# Deploy to VMs

Give LOKA your VM addresses and it will SSH in, install everything, and wire up the cluster.

## Single node

```bash
loka deploy vm --name dev --cp 10.0.0.1 --ssh-user ubuntu --ssh-key ~/.ssh/id_rsa
```

This SSHs into `10.0.0.1`, installs LOKA, starts the control plane and an embedded worker (using `role: all`).

## Multi-node

```bash
loka deploy vm --name prod \
  --cp 10.0.0.1 \
  --worker 10.0.0.2 \
  --worker 10.0.0.3 \
  --worker 10.0.0.4 \
  --ssh-user ubuntu \
  --ssh-key ~/.ssh/id_rsa
```

What happens:

1. SSHs into the CP node, installs LOKA, starts `lokad` with `role: controlplane` (CP only — no embedded worker)
2. Creates a worker registration token
3. SSHs into each worker node, installs LOKA, configures and starts `loka-worker`
4. Saves the deployment to `~/.loka/deployments.json`

## Options

| Flag | Default | Description |
|------|---------|-------------|
| `--name` | `vm` | Deployment name |
| `--cp` | | Control plane VM address (required) |
| `--worker` | same as CP | Worker VM addresses (repeatable) |
| `--ssh-user` | `root` | SSH username |
| `--ssh-key` | | SSH private key path |
| `--port` | `6840` | Control plane API port |

## Requirements

The VMs need:

- Linux with `/dev/kvm` (for workers)
- SSH access from your machine
- Internet access (to download the install script)

## What gets installed

On each node:

- `lokad` — control plane daemon
- `loka-worker` — worker agent
- `loka-supervisor` — in-VM supervisor
- `loka` — CLI
- Firecracker + Linux kernel

## Declarative deployment with a YAML file

Instead of CLI flags, you can describe your cluster in a YAML file and run `deploy apply`. This is useful when you want to version-control your infrastructure or manage workers over time.

```yaml
name: production
provider: vm
ssh:
  user: ubuntu
  key: ~/.ssh/id_rsa
controlplane:
  address: 10.0.0.1
workers:
  - address: 10.0.0.2
  - address: 10.0.0.3
  - address: 10.0.0.4
```

```bash
loka deploy apply cluster.yml
```

On first run this is equivalent to `loka deploy vm` with the same flags. On subsequent runs, LOKA diffs the worker list against the live cluster — adding new nodes and draining/removing ones no longer in the file. The control plane is not re-deployed on updates.

**Update workflow:**

1. Edit `cluster.yml` (add or remove worker addresses)
2. Run `loka deploy apply cluster.yml` — only the diff is applied

**Export existing deployment as YAML:**

```bash
loka deploy export prod > cluster.yml   # Save current state
# Edit cluster.yml, then re-apply
loka deploy apply cluster.yml
```

## After deployment

```bash
loka use prod
loka status
loka image pull python:3.12-slim
loka session create --image python:3.12-slim
```

## Manage

```bash
loka list                 # See all servers
loka deploy status prod   # Check health
loka deploy down prod     # Stop
loka deploy destroy prod  # Remove
```
