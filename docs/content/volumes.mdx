# Volumes

Volumes provide persistent storage for sessions and services. Data in a volume survives session restarts, redeployments, and can be shared across components.

LOKA supports two volume types:

| Type | Description | Replication | Use case |
|------|-------------|-------------|----------|
| **Block** | Folder on worker host, mounted via virtiofs | 2 replicas across workers | Application data, databases, uploads |
| **Object** | Backed by object storage (S3/GCS/Azure) | Object store is source of truth | Assets, media, shared files |

## Block volumes

Block volumes are the default. Data lives directly on the worker host as a regular directory, mounted into VMs via virtiofs for native filesystem performance. Each block volume is replicated to a second worker for durability.

### Create a block volume

```bash
loka volume create my-data
```

With a size limit:

```bash
loka volume create my-data --max-size 10G
```

### Replication

Every block volume maintains 2 copies across different workers:

- **Primary**: the worker running the service that mounts the volume
- **Replica**: a second worker holding a copy for failover

Replication is fully async -- writes are never blocked. Changes are detected via filesystem notifications and pushed to the replica worker in the background (typically within 200ms).

```
Write path:  VM → virtiofs → host filesystem → done (full disk speed)
Replication: fsnotify → 200ms debounce → async push to replica
```

If the primary worker dies, the replica is automatically promoted to primary and a new replica is assigned.

In single-worker mode, block volumes run in **degraded** mode (1 copy only). When a second worker joins, replication starts automatically.

### Shared access

Multiple service instances can mount the same block volume. Use the lock API to coordinate access:

```bash
# Acquire a shared (read) lock
curl -X POST http://localhost:8080/api/v1/volumes/my-data/lock \
  -d '{"path": "/", "worker_id": "w1", "exclusive": false, "ttl": 30}'

# Acquire an exclusive (write) lock
curl -X POST http://localhost:8080/api/v1/volumes/my-data/lock \
  -d '{"path": "/", "worker_id": "w1", "exclusive": true, "ttl": 30}'
```

Multiple workers can hold shared locks simultaneously. An exclusive lock blocks all other holders.

## Object volumes

Object volumes are backed by object storage. Two sub-modes:

### Loka-managed

When no bucket is specified, LOKA uses its configured object store as the backend. If no object store is configured, the volume falls back to block mode.

```bash
loka volume create my-assets --type object
```

### Direct connection

Connect directly to your own S3/GCS/Azure bucket:

```bash
loka volume create my-assets --type object \
  --bucket my-s3-bucket \
  --prefix assets/ \
  --region us-east-1
```

Workers cache object volume data locally for performance and sync changes back to the bucket. The object store is the source of truth -- no worker-to-worker replication is needed.

## Mount a volume in a service

In `loka.yaml`:

```yaml
name: my-app
components:
  api:
    image: python:3.12-slim
    command: uvicorn main:app --host 0.0.0.0 --port 8000
    port: 8000
    mounts:
      - volume: my-data
        path: /data
      - volume: db-data
        path: /var/lib/postgresql/data
        read_only: true
```

Volumes are created automatically if they do not already exist.

## Mount a volume in a session

```bash
loka session create --image python:3.12-slim --volume my-data:/data
```

## Mount cloud storage directly

For sessions, you can mount cloud storage directly without creating a volume:

```bash
loka session create --image python:3.12-slim \
  --mount s3://bucket@/data?access_key_id=AKIA...&secret_access_key=...
```

Sync data between the VM and cloud storage:

```bash
loka session sync <id> /data --direction push   # VM --> cloud
loka session sync <id> /data --direction pull   # cloud --> VM
```

## List volumes

```bash
loka volume list
```

## Inspect a volume

```bash
loka volume inspect my-data
```

Shows volume details including type, status, size, worker placement, and mount count.

## Delete a volume

```bash
loka volume delete my-data
```

Volumes that are currently mounted cannot be deleted.

## Volume status

| Status | Meaning |
|--------|---------|
| `healthy` | All replicas present, fully redundant |
| `degraded` | Fewer replicas than desired (e.g., single-worker mode) |
| `syncing` | Replica catch-up in progress |
| `error` | Data loss -- all copies lost |

## Scheduling affinity

The scheduler prefers workers that already hold a volume's primary copy. This avoids unnecessary data transfers when redeploying a service.
